## 2. 产品一句话

智慧排班 Agent 是面向零售门店的 AI 劳动力运营系统，基于销售、客流、订单、库存、促销、天气、员工技能、员工偏好和劳动合规规则，自动生成并动态调整“谁在什么时间、在哪个岗位、做什么任务”的门店排班和工作计划。

## 3. 背景与机会

### 3.1 行业背景

零售门店的经营高度依赖一线员工。门店需要在客流高峰、线上订单高峰、补货窗口、促销活动、鲜食制作、盘点和清洁等任务之间动态分配人力。传统排班通常由店长基于经验手工完成，主要问题包括：

- 排班耗时高，店长每周需要反复协调员工可用时间、请假、换班和临时缺勤。
- 排班与真实需求脱节，高峰期人手不足，低峰期人力闲置。
- 排班只解决“谁上班”，没有解决“上班后做什么”。
- 员工班表不可预测，影响满意度、收入稳定性和留任。
- 劳动法规对提前通知、班次变更、休息间隔、加班和记录留存的要求越来越细。

公开资料显示，零售劳动力管理已经从后台工具变成影响利润、服务体验和员工体验的核心运营能力：

- Deloitte 在零售劳动力管理观点中指出，门店劳动力是零售商最大的可控成本之一，标准化排班、自动排班和 AI 辅助劳动力优化可帮助零售商降低成本并改善客户体验。
- Logile 2025 年针对 500 名美国零售门店员工的调研显示，77% 的一线员工认为糟糕排班或人员配置会导致门店损失销售，80% 表示不可预测排班和人手不足会增加工作压力。
- 洛杉矶县 Fair Workweek Ordinance 已于 2025-07-01 生效，要求覆盖范围内的零售雇主提供可预测排班、班次变更权利和 predictability pay 等保护。
- 纽约市 Fair Workweek Law 要求零售雇主至少提前 72 小时提供工作排班，且限制 on-call 班次和临时取消班次。

### 3.2 用户痛点

#### 总部运营/人力规划团队

- 很难统一多门店排班标准。
- 无法准确评估门店人效差异来自客流、员工技能、店长经验还是排班质量。
- 劳动力预算与实际销售、客流、线上订单和促销活动脱节。
- 缺少跨门店人力调配机制。
- 无法提前发现合规风险和异常加班。

#### 区域经理

- 需要同时关注多个门店的人力缺口和服务质量，但缺少实时预警。
- 门店之间排班方式不一致，难以复制优秀门店经验。
- 临时支援主要依赖电话、群消息和人工协调。

#### 门店店长

- 每周排班需要反复处理员工偏好、请假、换班、技能覆盖和预算约束。
- 客流突增、员工迟到、外卖订单暴增、到货延迟时，需要临时重新安排岗位。
- 不知道某个时段应该优先保障收银、导购、补货、拣货、鲜食制作还是清洁。
- 调整排班后难以向员工解释原因。

#### 一线员工

- 班表不稳定，影响生活安排和收入预期。
- 换班、请假、抢班流程低效。
- 排班公平性不透明，容易产生不满。
- 技能成长与更多工时、更多岗位机会之间没有明确连接。

### 3.3 产品机会

现有 Workforce Management 系统通常覆盖排班、考勤、工时和合规，但零售门店仍存在三个未被充分解决的机会：

1. 从“班表”升级为“任务级劳动力计划”：不仅安排员工上班，还安排员工在具体时段负责具体岗位和任务。
2. 从“周排班”升级为“日内动态调整”：根据客流、订单、员工缺勤、天气和到货变化，实时给出调整建议。
3. 从“规则工具”升级为“可解释 Agent”：用自然语言解释排班理由、约束冲突、合规风险和员工偏好取舍。

## 4. 产品目标

### 4.1 业务目标

| 目标 | 说明 |
| --- | --- |
| 降低门店劳动力浪费 | 减少低峰冗员、无效工时和不必要加班 |
| 提升高峰服务能力 | 在高峰期确保关键岗位覆盖，减少排队、缺货和订单延误 |
| 提升店长效率 | 将排班从经验型、手工型工作转为系统推荐和人工审批 |
| 提升员工体验 | 提供稳定班表、偏好设置、换班申请和公平解释 |
| 降低合规风险 | 在排班生成和变更前自动校验劳动法规和企业规则 |

### 4.2 MVP 量化目标

| 指标 | 目标值 |
| --- | --- |
| 店长每周排班耗时 | 降低 50% 以上 |
| 高峰期关键岗位缺口 | 降低 20% 以上 |
| 计划工时与实际需求偏差 | 降低 10%-15% |
| 店长对系统推荐班表采纳率 | 大于 60% |
| 员工端查班/换班/请假使用率 | 大于 70% |
| 班表发布提前天数 | 达到企业或当地法规要求 |
| 合规风险拦截率 | 重大违规发布前 100% 提醒 |

## 6. 用户画像

### 6.1 总部劳动力规划经理

- 关注目标：控制人力成本、提升人效、统一排班标准、减少合规风险。
- 使用频率：每周查看计划，每日查看异常。
- 核心问题：哪些门店人力过剩或不足？哪些门店需要调整预算？哪些规则经常冲突？

### 6.2 区域经理

- 关注目标：区域门店服务水平、销售达成、跨店支援。
- 使用频率：每日。
- 核心问题：今天哪些门店高峰缺人？是否可以从附近门店调人？哪个店长需要干预？

### 6.3 门店店长

- 关注目标：快速排好班、现场执行顺畅、减少员工投诉。
- 使用频率：每日。
- 核心问题：这周怎么排？明天谁请假了？晚高峰谁去收银？到货后谁补货？

### 6.4 一线员工

- 关注目标：提前知道班表、申请偏好、换班、获得稳定工时。
- 使用频率：每日到每周。
- 核心问题：我什么时候上班？能不能换班？为什么这周我的工时变少？我还能抢哪些班？

## 7. 核心场景

### 7.1 周排班

1. 系统同步未来 1-4 周的销售预测、客流预测、促销计划、线上订单预测和员工可用时间。
2. 系统预测每个门店、每个时段、每个岗位所需人数或工时。
3. 系统根据预算、员工技能、合同工时、偏好和法规生成初版班表。
4. 店长查看冲突、风险和推荐理由。
5. 店长手动调整或通过 Agent 发起调整。
6. 系统二次校验后发布班表。
7. 员工收到通知并确认。

### 7.2 日内动态调整

1. 系统监测到客流、销售、线上订单、员工缺勤或到货计划出现异常。
2. Agent 判断影响范围，例如“17:00-19:00 收银缺口 1 人，补货任务可延后”。
3. Agent 给出调整方案，例如“将员工 A 从补货调整到收银，员工 B 延后补货，邀请员工 C 加班 2 小时”。
4. 店长确认或修改。
5. 系统通知相关员工并记录变更原因。

### 7.3 员工换班

1. 员工发起换班申请。
2. 系统筛选符合技能、工时、可用性和合规要求的候选人。
3. 候选员工接受后，系统提交店长审批或按企业规则自动审批。
4. 系统更新班表并通知相关人员。

### 7.4 合规拦截

1. 店长手动调整班表。
2. 系统检测到员工连续工作间隔不足、超出工时上限或违反提前通知规则。
3. 系统阻止发布或提示风险等级。
4. Agent 给出可替代方案。

## 10. 产品架构

### 10.1 核心模块

```text
数据接入层
  POS / 客流 / 订单 / 库存 / 促销 / 天气 / 员工 / 考勤

预测层
  客流预测 / 销售预测 / 订单预测 / 岗位需求预测

优化层
  约束规则 / 班表优化 / 任务分配 / 跨店支援推荐

Agent 层
  自然语言理解 / 排班解释 / 调整建议 / 冲突说明

应用层
  总部看板 / 区域看板 / 店长工作台 / 员工端

治理层
  权限 / 审计 / 合规 / 数据安全 / 指标监控
```

### 10.2 推荐技术路线

- 预测模型：时间序列模型 + 机器学习特征模型，用于预测销售、交易数、客流和订单量。
- 工时换算模型：根据交易数、补货件数、订单量和任务标准工时，换算岗位级工时需求。
- 优化引擎：使用约束规划、整数规划或启发式算法，在预算、技能、偏好、法规和公平性之间求解。
- LLM Agent：负责自然语言交互、解释、方案摘要和异常处理建议。
- 规则引擎：承载企业规则和地区劳动法规，避免将合规规则硬编码。
- 人工审批：所有影响员工收入、工时或合规的重大变更必须由授权人员确认。

| 考勤数据 | 实际上班时间、实际下班时间、迟到、早退、缺勤、加班 |
| 促销数据 | 活动日期、活动类型、参与品类、预期影响 |
| 规则数据 | 最小休息间隔、最大连续工作时长、提前通知、加班规则 |

### 11.2 可选增强数据

| 数据 | 用途 |
| --- | --- |
| 客流摄像头/客流计数器 | 提升客流预测准确度 |
| 天气 | 影响到店客流和即时零售订单 |
| 线上订单 | 预测拣货、打包和交付任务 |
| 库存和到货计划 | 预测补货、上架和盘点工时 |
| 商圈活动 | 预测异常高峰 |
| 员工满意度 | 优化公平性和留任风险 |

## 14. 成功指标

### 14.1 北极星指标

门店有效覆盖率 = 实际关键岗位覆盖时长 / 系统预测关键岗位需求时长

该指标同时反映需求预测、排班质量和现场执行质量。

### 14.2 业务指标

- 每工时销售额。
- 人力成本占销售额比例。
- 高峰期岗位缺口率。
- 收银排队时长。
- 线上订单履约及时率。
- 缺货补货及时率。

### 14.3 产品指标

- 自动班表生成次数。
- 推荐班表采纳率。
- 店长手动修改率。
- Agent 调整请求成功率。
- 员工端活跃率。
- 换班线上完成率。

### 14.4 员工体验指标

- 班表提前发布天数。
- 员工偏好满足率。
- 周工时稳定性。
- 换班成功率。
- 缺勤率。
- 员工满意度。

### 14.5 模型指标

- 客流预测 MAPE。
- 交易量预测 MAPE。
- 岗位工时预测误差。
- 高峰缺口预测准确率。
- 异常预警准确率。

## 15. MVP 版本计划

### 阶段 0：客户验证，2-3 周

- 访谈 5-10 家零售企业。
- 访谈总部运营、区域经理、店长和一线员工。
- 收集样例数据：销售、排班、考勤、员工档案。
- 明确首个垂直业态，例如便利店或生鲜商超。
- 输出业务规则清单和 MVP 试点范围。

### 阶段 1：MVP 开发，8-10 周

- 完成数据接入。
- 完成基础需求预测。
- 完成自动排班引擎。
- 完成店长工作台。
- 完成员工端基础功能。
- 完成基础规则校验。
- 完成 Agent 解释和调班能力。

### 阶段 2：试点运行，8-12 周

- 选择 20-50 家门店灰度。
- 与现有人工排班并行 2 周。
- 逐步切换到系统推荐、店长审批发布。
- 每周复盘预测误差、采纳率、缺岗率、员工反馈。
- 达成试点门槛后进入规模化推广。

### 阶段 3：规模化，3-6 个月

- 支持跨门店支援。
- 支持日内动态重排。
- 支持任务完成反馈。
- 支持更复杂合规规则。
- 支持总部级预算模拟和策略下发。

## 16. 商业化设想

### 16.1 目标客户

优先选择以下客户：

- 门店数 20 家以上。
- 员工排班复杂，有兼职、全职、多技能、多岗位。
- 客流和订单波动明显。
- 店长人工排班负担高。
- 已有 POS、考勤或基础人事数据。

### 16.2 计费方式

可采用 SaaS 订阅模式：

- 按门店/月计费。
- 按员工/月计费。
- 高级模块单独计费，例如跨店调度、日内动态重排、合规包、预测增强包。
- 企业客户可按总部项目制 + 年费。

### 16.3 客户 ROI 逻辑

ROI 来自：

- 减少排班管理时间。
- 降低冗余工时和加班。
- 提升高峰服务能力，减少销售损失。
- 降低员工流失和临时缺勤。
- 减少合规罚款和争议。

## 17. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 数据质量不足 | 预测不准，店长不信任 | 先做数据健康检查，允许店长修正，持续回流实际结果 |
| 店长抵触 | 采纳率低 | 保留人工调整权，提供解释和方案对比 |
| 员工抵触 | 使用率低，投诉增加 | 提供偏好设置、公平性解释和稳定发布机制 |
| 合规复杂 | 违规风险 | 使用规则引擎，按地区配置，发布前强校验 |
| 预测误差 | 高峰缺人或低峰冗员 | 输出置信度，设置备班池和异常预警 |
| 竞品压力 | 差异化不足 | 聚焦任务级规划、日内动态调整和可解释 Agent |
| 过度自动化 | 影响员工收入和信任 | 重大变更必须人工审批，所有变更留痕 |

## 18. 待验证问题

1. 首个落地业态应选择便利店、生鲜商超、服饰零售还是药店？
2. 客户是否已有可用的历史排班、考勤和 POS 数据？
3. 店长当前每周排班实际耗时是多少？
4. 高峰期缺人对销售、排队和订单履约的影响能否量化？
5. 员工是否愿意维护可用时间和班次偏好？
6. 企业是否允许系统推荐跨店支援？
7. 首个试点地区是否涉及 Fair Workweek、提前通知或其他预测性排班法规？

## 19. 立项建议

建议立项，但产品切入点不应是普通“AI 自动排班工具”。更有价值的产品形态是“零售门店劳动力运营 Agent”：

- 用预测模型回答“什么时候需要多少人”。
- 用任务模型回答“这些人应该做什么”。
- 用优化引擎回答“谁最适合做”。
- 用 Agent 回答“为什么这样排、如何调整、风险在哪里”。
- 用员工端回答“员工如何参与、确认、换班和表达偏好”。

MVP 应优先选择数据基础较好、门店任务标准化程度较高、排班痛点强的连锁便利店或生鲜商超作为试点客户。
