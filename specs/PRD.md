## 1. 背景与问题

现有门店排班通常以区域为单位：水产区、肉类区、果蔬区、收银区、补货区分别安排员工。该方式可以保障专业岗位稳定，但在实际经营中暴露出三个问题：

| 编号 | 问题 | 描述 |
| --- | --- | --- |
| P-01 | 系统推荐后仍需要大量人工调整 | 店长经常根据"谁是师傅""谁不能离开本区域""谁可以跨区帮忙"等经验改班，说明系统没有充分理解门店真实用人规则 |
| P-02 | 区域固定导致低峰人力利用不足 | 某些区域低峰有人闲置时，其他区域可能出现补货、称重、收银、打包等通用任务缺口 |
| P-03 | 全员混排不可接受 | 杀鱼、切肉、熟食加工等岗位依赖专业技能和稳定人员，不能简单按照"谁有空就排谁"分配 |

因此，本 Demo 采用**半混班**方案：专业岗位固定保障，通用岗位跨区补位。

---

## 2. 版本结论

本 Demo 只验证一个核心命题：

> 在生鲜商超门店中，系统先稳定锁定水产、肉类等专业区域的关键师傅，再使用全店混排池补齐高峰和通用任务缺口，从而减少店长手动改班，并提升通用人力利用率。

本版本聚焦单门店半混班智能排班场景，核心目标是验证 **Agent 能基于历史经营数据、高峰/低峰规律、节假日、天气等因素综合估算用工需求，并在半混班约束下生成最优或近似最优排班方案**。

研发实现应优先保证三件事：

1. **需求计算可信**：能说明为什么某天某时段需要更多或更少人。
2. **排班结果可用**：专业岗稳定，通用人力能跨区补位。
3. **Agent 可解释**：能解释预测依据、排班取舍、不可调原因和替代方案。

## 3. 产品定位

### 3.1 产品一句话

智慧排班 Agent 半混班 Demo 是一个面向生鲜商超门店的排班工作台：Agent 基于历史销售、客流、订单、节假日、天气等因素计算未来 7 天分时段用工需求，再在专业岗稳定、混排池补位、区域保底和工时约束下生成半混班班表，并解释每一次需求判断、固定、混排和不可调整原因。

### 3.2 Demo 成功标准

| 编号 | 标准 | 说明 |
| --- | --- | --- |
| S-01 | 能生成 1 家门店 7 天半混班班表 | 完整输出一周排班 |
| S-02 | 专业固定岗优先由专业人员覆盖 | 杀鱼、切肉等岗位由 S/A 级师傅承担 |
| S-03 | 高峰缺口可优先从混排池补位 | 不轻易打乱专业固定岗 |
| S-04 | 系统能解释为什么固定、为什么混排、为什么不可调 | 每个决策都有可追溯的解释 |
| S-05 | 店长人工修改能被记录、归因和统计 | 记录干预率和高频原因 |
| S-06 | 5 分钟内可完整演示闭环 | 从需求计算→生成班表→发现缺口→Agent 推荐→解释不可调→记录干预 |

## 5. 关键概念

| 概念 | 定义 | 示例 |
| --- | --- | --- |
| 正式工 (Regular) | 固定在特定区域工作，排班只在所在部门，不跨区支援 | 老王(水产)、老张(肉类)、小李(果蔬) |
| 临时工 (Temporary) | 跨部门混班排班，不干专业工作，每班3-6小时，根据忙碌程度决定 | 小唐、小马 |
| 正式工早班 | 正式工班次之一，时段 8:00-16:00 | 水产区老王排早班8:00-16:00 |
| 正式工晚班 | 正式工班次之二，时段 14:00-22:00 | 收银小吴排晚班14:00-22:00 |
| 正式工两头班 | 正式工班次之三，时段 8:00-12:00,17:00-21:00（中间休息） | 果蔬区老宋排两头班8:00-12:00,17:00-21:00 |
| 正式工周意愿 | 正式工每周可申请变动下周班次类型，如本周早班可申请下周晚班或两头班 | 老宋本周两头班，申请下周早班 |
| 专业固定岗 | 需要专业资质或高熟练度，仅由具备资质的正式工独立承担 | 杀鱼、切肉 |
| 区域专业人士 | 长期稳定服务某区域，具备专业固定岗能力的正式工 | 水产师傅、肉类师傅 |
| 临时工排班 | 临时工跨部门混排，工作日固定18:00-21:00、周末根据忙碌程度决定时长（无"每部门每天最多1班"约束） | 果蔬区周五安排小唐16:00-18:00帮忙 |
| 全店混排池 | 可跨区域承担通用任务的临时工集合 | 小唐、小马 |
| 混排任务 | 不强依赖专业固定岗能力，可由临时工承担的任务 | 补货、陈列、打包、称重 |
| 区域保底正式工人数 | 每个区域每个班次必须保留的最低正式工人数 | 水产区早班至少2人 |
| 部门临时工名额 | 无此约束（临时工可跨部门多班混排，已移除"每部门每天最多1班"限制） | — |
| 人工干预 | 店长对系统推荐班表进行的新增、删除、替换、调时、改区域操作 | 把小唐从果蔬改到收银 |
| 正式工每周休假 | 正式工每周有 1 天假期，可自行选择星期几 | 老王申请周三休假 |
| 休假冲突 | 同日休假人数过多导致该日人手不足，系统自动驳回部分人的申请 | 周一有 8 人申请休假，系统批准其中 5 人、驳回 3 人 |

## 7. Demo 业务流程

### 7.1 主流程

1. 店长进入半混班排班工作台。
2. 系统加载样例门店、区域、岗位、员工、技能、需求数据。
3. 店长点击"生成下周半混班班表"。
4. 系统先锁定专业固定岗和保护时段。
5. 系统校验各区域保底人数。
6. 系统生成全店混排池。
7. 系统按高峰缺口和任务优先级进行混排补位。
8. 系统输出周班表、区域覆盖、风险提示、KPI 和解释。
9. 店长查看缺口或风险，可点击 Agent 提问。
10. 店长进行人工修改时，系统记录修改项和原因。
11. 系统更新干预率、风险和相关解释。

### 7.2 核心演示路径

| 步骤 | 用户操作 | 系统反馈 |
| --- | --- | --- |
| 1 | 点击"生成下周半混班班表" | 生成 7 天班表，展示专业固定岗和混排岗 |
| 2 | 查看周五晚高峰 | 展示果蔬/收银缺口和可补位候选人 |
| 3 | 向 Agent 提问"周五晚高峰果蔬缺人，谁能支援" | 推荐小唐、小马等候选人，并解释推荐原因 |
| 4 | 向 Agent 提问"能不能把老张调去收银" | 说明老张为肉类切肉师傅，保护时段不能抽调 |
| 5 | 手动修改一条排班 | 弹出修改原因，记录干预并更新人工干预率 |
| 6 | 查看 KPI | 展示专业岗覆盖率、区域保底达成率、混排利用率、人工干预率 |

## 9. 业务规则

### 9.1 规则优先级

| 优先级 | 规则类型 | 处理方式 |
| --- | --- | --- |
| 1 | 正式工部门归属 | 正式工只在所属部门排班，不跨区 |
| 2 | 专业岗资质 | 无资质正式工不得独立承担专业固定岗 |
| 3 | 部门保底正式工 | 每个部门各班次必须满足最低正式工保底人数，无法满足则标记风险 |
| 4 | 临时工时长度 | 临时工工作日固定 18:00-21:00（3小时）；周末 3-6 小时按部门忙碌程度决定 |
| 5 | （R-005 已废弃/合并）部门临时工名额 | 原"每部门每天最多1班"约束已移除，临时工可跨部门多班混排，无此限制 |
| 6 | 员工可用性 | 不在可用时段内的员工不推荐排班（含休假员工） |
| 7 | 正式工每周休假 | 正式工每周 1 天假期，可自行选择星期几；系统确保每日在岗人数满足保底要求 |
| 8 | 休假冲突解决 | 同日休假超过允许上限时，系统自动驳回超出部分，优先保留先申请的员工 |
| 9 | 周工时上限 | 超过周工时上限的候选人直接不进入排班（硬约束），无扣分 |
| 10 | 临时工技能匹配 | 临时工必须具备目标任务最低技能，不得安排专业任务 |
| 11 | 区域熟悉度 | 曾在该区域工作过的临时工优先 |
| 12 | 工时均衡 | 当前周工时较低者优先 |
| 13 | 员工偏好 | 满足偏好者优先 |

### 9.2 硬规则

| 编号 | 规则 | 说明 | 失败处理 |
| --- | --- | --- | --- |
| R-001 | 正式工固定在所属部门排班 | 正式工不跨区，排班范围限于其 main_area | 标记严重风险，不推荐跨区 |

| R-004 | 临时工工作日固定 18:00-21:00（3小时）；周末 3-6 小时按部门忙碌程度决定 | 工作日固定3小时，周末按忙碌程度确定3-6小时 | 调整时长至合法范围 |

| R-007 | 正式工每周休假一天 | 正式工每周可申请 1 天休假，批准后当天不排班 | 标记休假冲突风险，驳回超限申请 |
| R-008 | 周工时不能超过上限 | 每名员工配置周工时上限（正式工默认 48 小时） | 硬约束：超过周工时上限的候选人直接不进入排班，无扣分 |

## 11. HC 优化与人力成本

### 11.1 功能概述

HC 优化（编制优化）能力以**降本增效**为目标：在保障服务水平（专业岗覆盖 100%、区域保底满足、风险可控、需求覆盖率 ≥ 阈值如 0.95）的前提下，根据历史需求（`demand_results`）与实际排班（`schedule_items`）数据，动态建议正式工 / 临时工编制调整，在服务水平与人力成本之间求平衡（成本服务平衡），从而降低人力成本。系统输出分区域、分员工类型（`regular` 正式工 / `temporary` 临时工）的编制调整建议，并经人工确认后生效。

### 11.2 成本模型与数据依据

| 项目 | 说明 |
| --- | --- |
| 正式工周成本权重 C_reg | 1.0，按人数计，固定 |
| 临时工周成本权重 C_temp | 0.4，按实际使用工时计，弹性 |
| 周成本估算 | ≈ Σ(regular_count × C_reg) + Σ(temp_assigned_hours × C_temp)，权重可配 |
| 数据依据 | 历史 `demand_results`（需求）与 `schedule_items`（实际排班）对比，按区域 + 技能统计 `horizon_weeks`（默认 12 周）内的需求覆盖率、长期缺人时段占比、长期冗余工时，推导编制调整方向 |

### 11.3 优化逻辑

优化逻辑与场景判定（长期缺人/冗余、成本 delta 比较）见 [SPEC.md §11.3]。

### 11.4 功能需求列表

| 编号 | 需求 | 优先级 | 交互/输出 | 验收标准 |
| --- | --- | --- | --- | --- |
| FR-601 | 生成 HC 优化建议 | P0 | Agent 工具 `hc_optimize(horizon_weeks?: int)` 与 `generate_schedule` 等并列；`POST /api/hc/optimize` 返回 `HcOptimizeResult` | 建议须**区分正式工 / 临时工**，给出每区域每类型的 `current_count`、`suggested_count`、`delta` 及 `reason`；须提供 `est_cost_before` 与 `est_cost_after` 的**成本前后对比**；默认 `horizon_weeks=12` 可配 |
| FR-602 | 人工确认生效 / 拒绝建议 | P0 | `GET /api/hc/suggestions`（列表含 status）；`POST /api/hc/suggestions/confirm`（body `{suggestion_ids:[], action:'apply'\|'reject'}`） | 建议默认 `status='pending'`；`confirm→'approved'` 生效（调整员工编制 / 种子），`reject→'rejected'`；**必须人工确认才生效**，系统不得自动应用 |
| FR-603 | HC 成本趋势看板 | P0 | 看板展示各区域、各员工类型建议前后成本、总成本 `total_cost_before/after` 与 `summary` | 看板按正式工 / 临时工分别展示成本前后对比与 `delta`；可查看 `pending/approved/rejected` 状态分布 |

### 11.5 输出结构与状态机

输出结构（`HcSuggestion` / `HcOptimizeResult`）见 [SPEC.md §11.4]；状态机 `pending→approved/rejected`（人工 confirm 生效、reject 驳回，绝不自动应用）。

### 11.6 Agent 工具与接口

Agent 工具 `hc_optimize` 与接口见 [API.md §3.4] 与 [SPEC.md §6.2]。

### 11.7 数据表

存储表 `hc_suggestions` 见 [数据库表结构设计.md §3.18]。

### 11.8 用户故事示例

> 店长每月查看系统生成的 HC 优化建议：系统按区域对比近 12 周历史需求与排班，指出水产区晚高峰长期缺人、补货区周末存在长期冗余工时。建议中区分正式工（核心缺口增 1 名正式工）与临时工（波峰增 2 名、冗余区减 3 名小时池），并给出每区域成本前后对比（预计周成本由 142 降至 128）。店长确认水产区增编、补货区减临时工弹性池后，系统将建议置为 `approved` 并调整员工编制与排班种子，人力成本下降且服务水平未受影响。

## 15. KPI 与计算口径

| 指标 | 计算公式 | Demo 目标 |
| --- | --- | --- |
| 需求解释覆盖率 | 可解释需求时段数 / 关键需求时段数 | 覆盖高峰、低峰、节假日、天气四类 |
| 专业岗覆盖率 | 已由 S/A 正式工覆盖的专业岗需求数 / 专业岗总需求数 | 100% |
| 部门保底达成率 | 达成保底正式工人数的部门班次数 / 部门班次总数 | ≥ 90% |
| 临时工利用率 | 临时工支援工时 / 总排班工时 | 10%-25% |
| 高峰缺口数 | 高峰时段未满足需求人数之和（含正式工和临时工缺口） | 比纯区域固定排班少 |
| 人工干预率 | 被人工修改的排班项数 / 系统生成排班项数 | 可记录、可解释 |
| Agent 解释覆盖率 | 可被 Agent 解释的关键排班/风险数 / 关键排班/风险总数 | 覆盖正式工排班、临时工排班、不可调三类 |

| 不可调解释 | 尝试抽调专业师傅时能解释原因并给替代方案 |
| 干预记录 | 修改班次后必须记录修改项和原因 |
| KPI 展示 | 展示专业岗覆盖率、区域保底达成率、混排利用率、人工干预率、高峰缺口数 |
| Agent 解释 | 支持固定岗、混排、不可调、风险、减少干预五类问题 |

### 17.2 演示验收

5 分钟内必须完成以下演示：

1. 打开半混班排班工作台。
2. 点击生成下周半混班班表。
3. 展示 Agent 对历史数据、高峰/低峰、节假日、天气等因素的需求计算结果。
4. 展示专业岗锁定和混排标签。
5. 定位周五晚高峰果蔬/收银缺口。
6. Agent 推荐支援人员。
7. Agent 解释为什么不能抽调老张。
8. 手动修改一条排班并选择原因。
9. 展示人工干预率和干预原因。

## 19. 边界场景

| 场景 | 系统处理 |
| --- | --- |
| 历史数据不足 | 使用样例基线和规则兜底，并提示置信度较低 |
| 天气/节假日数据缺失 | 使用历史同星期/同时间段基线计算，并提示缺失因素 |
| 专业师傅不足 | 标记严重风险，说明缺少 S/A 专业人员 |
| 混排池无人可用 | 展示无候选原因：技能不满足、超工时、影响主区域保底等 |
| 候选人技能较弱 | 可推荐但标记黄色提醒 |
| 调走专业师傅会破坏保底 | 拒绝推荐，并给混排池替代方案 |
| 手动修改导致超工时 | 标记风险并要求填写原因 |
| 同一员工跨区过多 | 标记提示，建议减少跨区频次 |
| 用户提问超出 Demo 范围 | Agent 说明当前 Demo 暂不支持，并引导到支持问题 |

## 20. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 规则表达不清 | 研发实现口径不一致 | 以本 PRD 的规则优先级和验收标准为准 |
| 样例数据不足 | 演示看不出混排价值 | 必须内置周五晚高峰缺口和不可调案例 |
| LLM 输出不稳定 | 解释不可信或建议越界 | LLM 只做编排和解释，排班结果必须经过规则校验，关键指标由程序计算 |
| 需求计算依据不足 | 排班看起来像写死 | 页面展示历史数据、峰谷、节假日、天气等需求因素 |
| 页面信息过载 | 5 分钟演示难以讲清 | 主页面只保留需求洞察、KPI、区域、周班表、热力图、Agent 六块 |
| 算法结果不稳定 | 每次演示结果不一致 | Demo 使用确定性规则和固定样例数据 |
