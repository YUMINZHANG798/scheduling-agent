## 1. 设计总览

### 1.1 设计目标

| 目标 | 说明 |
| --- | --- |
| 支持单门店半混班排班闭环 | 存储需求计算、排班生成、风险检测、人工干预全流程数据 |
| 可重置可复现 | Demo 数据可一键重置，相同输入产生相同输出 |
| 可解释性支撑 | 每个排班项和风险都能追溯到员工、规则和数据 |
| 适度冗余提升查询性能 | 姓名等常用字段冗余存储，避免频繁 JOIN |

### 1.2 存储策略

| 数据类型 | 存储媒介 | 生命周期 | 说明 |
| --- | --- | --- | --- |
| 静态种子数据 | CSV / JSON 文件 | 持久（只读） | 区域、员工、技能等配置由文件加载到数据库 |
| 运行时数据 | SQLite 表 | Demo 会话 | 需求结果、排班、风险、干预记录可被重置清空 |
| 字典数据 | SQLite 表（预置） | 持久（只读） | 修改原因等字典数据 |

### 1.3 数据库文件

```
路径: backend/app/data/demo.sqlite
初始化: 首次启动或重置 Demo 时自动创建
重置: POST /api/demo/reset 清空运行时表，重新加载种子数据
```

### 3.4 employees — 员工信息表

员工分为**正式工 (regular)** 和 **临时工 (temporary)** 两类：
- **正式工**固定在特定区域工作，排班只在所在部门，三班：早班 8:00-16:00、晚班 14:00-22:00、两头班 8:00-12:00+17:00-21:00（中间休息）
- **临时工**可以跨部门混班，不干专业工作，工作日固定 18:00-21:00（3 小时），周末根据忙碌程度决定时长和班次

**种子数据共 90 人：正式工 80 人 + 临时工 10 人。分布如下：**

| 区域 | 正式工 | 临时工 | 合计 |
| --- | --- | --- | --- |
| 水产区 | 16 | 2 | 18 |
| 肉类区 | 16 | 2 | 18 |
| 果蔬区 | 24 | 3 | 27 |
| 收银/前场 | 16 | 2 | 18 |
| 补货区 | 8 | 1 | 9 |

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | TEXT | PK | — | 员工 ID，格式 `emp_001` |
| name | TEXT | NOT NULL | — | 员工姓名 |
| main_area | TEXT | FK → areas(code) | — | 主区域编码（正式工必填，临时工可为 NULL） |
| employee_type | TEXT | NOT NULL, CHECK('regular','temporary') | — | 员工类型：正式工/临时工 |
| regular_shift_type | TEXT | CHECK('morning','evening','split','rotating') | — | 正式工默认班次类型：早班/晚班/两头班/轮班；临时工该字段为 NULL |
| weekly_hours_limit | INTEGER | NOT NULL, > 0 | 48 | 周工时上限（小时），正式工每周 6 天 × 8h = 48h，临时工根据合同确定 |
| is_active | INTEGER | NOT NULL, CHECK(0 or 1) | 1 | 是否在职 |

> 注：`regular_shift_type` 为员工**默认**班次类型。员工每周可通过 `employee_weekly_preferences` 表申请变更下周班次意愿。调度时优先采用周意愿，若无则使用默认班次。正式工固定在所属部门排班（不外派跨区），临时工天然参与混班。该逻辑由 `employee_type` 隐式派生。

**索引：**

```sql

CREATE INDEX idx_employees_type ON employees(employee_type);
CREATE INDEX idx_employees_active ON employees(is_active);
```

### 3.6 employee_weekly_preferences — 员工周意愿表

记录员工每周的班次意愿。正式工可以每周申请变动下周的班次类型（如本周早班，申请下周晚班或两头班）。调度时优先采用周意愿，若本周无意愿记录则使用 `employees.regular_shift_type` 默认班次。

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | TEXT | PK | — | 唯一标识 |
| employee_id | TEXT | FK → employees(id), NOT NULL | — | 员工 ID |
| week_start | TEXT | NOT NULL | — | 意愿适用的排班周起始日期 YYYY-MM-DD（周一） |
| preferred_shift_type | TEXT | NOT NULL, CHECK('morning','evening','split') | — | 意愿班次类型：早班/晚班/两头班 |
| created_at | TEXT | NOT NULL | — | 创建时间 ISO 8601 |

> 注：`preferred_shift_type` 不含 `rotating`，因为员工申请应明确表达具体班次偏好。若员工无意愿则使用默认班次（可能为 `rotating` 由系统决定）。

**索引：**

```sql
CREATE INDEX idx_emp_weekly_prefs_employee ON employee_weekly_preferences(employee_id);
CREATE UNIQUE INDEX idx_emp_weekly_prefs_unique ON employee_weekly_preferences(employee_id, week_start);
```

### 3.7 employee_weekly_leave — 员工周休假意愿表

记录正式工每周的休假意愿。正式工每周可申请 1 天假期，自行选择星期几休息。

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | TEXT | PK | — | 唯一标识 |
| employee_id | TEXT | FK → employees(id), NOT NULL | — | 员工 ID |
| week_start | TEXT | NOT NULL | — | 意愿适用的排班周起始日期 YYYY-MM-DD（周一） |
| preferred_day_off | TEXT | NOT NULL, CHECK('Monday','Tuesday','Wednesday','Thursday','Friday','Saturday','Sunday') | — | 希望休假的那一天 |
| created_at | TEXT | NOT NULL | — | 创建时间 ISO 8601 |

> **冲突解决：** 排班生成时，系统统计每天有多少正式工申请休假。如果某天超过该天允许的最大休假人数（基于当天各区域保底需求计算出最低在岗人数），则**部分人的申请被驳回**（`leave_resolution` 表记录最终结果），被驳回的员工仍需正常排班。

**索引：**

```sql
CREATE INDEX idx_emp_weekly_leave_employee ON employee_weekly_leave(employee_id);
CREATE INDEX idx_emp_weekly_leave_week ON employee_weekly_leave(week_start);
CREATE UNIQUE INDEX idx_emp_weekly_leave_unique ON employee_weekly_leave(employee_id, week_start);
```

### 3.8 leave_resolution — 假期冲突解决表

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | TEXT | PK | — | 唯一标识 |
| version_id | TEXT | FK → schedule_versions(id), NOT NULL | — | 排班版本 ID |
| employee_id | TEXT | FK → employees(id), NOT NULL | — | 员工 ID |
| day_off | TEXT | NOT NULL, CHECK('Monday','Tuesday','Wednesday','Thursday','Friday','Saturday','Sunday') | — | 最终休假日（即使驳回也可记录原意愿） |
| status | TEXT | NOT NULL, CHECK('approved','denied') | 'approved' | 批准/驳回 |
| reason | TEXT | — | — | 驳回原因，如"周一休假申请超限，系统已驳回" |

**索引：**

```sql
CREATE INDEX idx_leave_resolution_version ON leave_resolution(version_id);
CREATE INDEX idx_leave_resolution_employee ON leave_resolution(employee_id);
```

### 3.11 schedule_items — 排班项表

排班结果的核心表，每条记录为一个员工在一个时段的一个任务分配。

- **正式工**：`slot` 填写其完整班次，如 `08:00-16:00`（早班）、`14:00-22:00`（晚班）或 `08:00-12:00,17:00-21:00`（两头班），`assignment_type = 'regular'`
- **临时工**：工作日固定 `18:00-21:00`（3 小时），周末根据忙碌程度决定时长，如 `09:00-14:00`、`16:00-18:00`，`assignment_type = 'temporary'`

> **slot 映射规则：** `demand_results.slot` 为 1 小时间隔（`08:00-09:00` ~ `21:00-22:00`），`schedule_items.slot` 为员工实际班次时段。排班时将 slot 级需求按班次聚合：
> - 早班 `08:00-16:00` = 08-09 + 09-10 + 10-11 + 11-12 + 12-13 + 13-14 + 14-15 + 15-16（8 个 slot）
> - 晚班 `14:00-22:00` = 14-15 + 15-16 + 16-17 + 17-18 + 18-19 + 19-20 + 20-21 + 21-22（8 个 slot）
> - 两头班 `08:00-12:00,17:00-21:00` = 08-09→11-12（4 个 slot）+ 17-18→20-21（4 个 slot）
> - 临时工工作日 `18:00-21:00` = 18-19 + 19-20 + 20-21（3 个 slot）

| 字段名 | 类型 | 约束 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | TEXT | PK | — | 排班项 ID，格式 `si_{uuid_short}` |
| version_id | TEXT | FK → schedule_versions(id), NOT NULL | — | 排班版本 ID |
| date | TEXT | NOT NULL | — | 日期 |
| slot | TEXT | NOT NULL | — | 时段（正式工为完整班次如 `08:00-16:00`，临时工为实际时段如 `09:00-14:00`） |
| area_code | TEXT | FK → areas(code), NOT NULL | — | 区域编码 |
| task_code | TEXT | FK → area_tasks(area_code, task_code), NOT NULL | — | 任务编码（临时工只分配非专业任务） |
| employee_id | TEXT | FK → employees(id), NOT NULL | — | 员工 ID |
| employee_name | TEXT | NOT NULL | — | 员工姓名（冗余，方便展示） |
| assignment_type | TEXT | NOT NULL, CHECK('regular','temporary') | — | 分配类型：正式工排班/临时工排班 |
| risk_level | TEXT | NOT NULL, CHECK('none','info','warning','critical') | 'none' | 风险等级 |
| explanation | TEXT | — | — | 排班解释 |
| source | TEXT | NOT NULL, CHECK('system','manual') | 'system' | 来源 |

PRAGMA foreign_keys = ON;

-- ============================================
-- 4.1 区域配置表
-- ============================================
CREATE TABLE IF NOT EXISTS areas (
    code TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    allow_mixed INTEGER NOT NULL CHECK(allow_mixed IN (0, 1)) DEFAULT 1,
    baseline_min INTEGER NOT NULL CHECK(baseline_min >= 0) DEFAULT 2,
    baseline_max INTEGER NOT NULL CHECK(baseline_max >= baseline_min) DEFAULT 3,
    sort_order INTEGER NOT NULL DEFAULT 0
);

-- 4.2 区域任务配置表
-- ============================================
CREATE TABLE IF NOT EXISTS area_tasks (
    id TEXT PRIMARY KEY,
    area_code TEXT NOT NULL REFERENCES areas(code) ON DELETE CASCADE,
    task_code TEXT NOT NULL,
    task_name TEXT NOT NULL,
    is_professional INTEGER NOT NULL CHECK(is_professional IN (0, 1)) DEFAULT 0,
    min_skill_level TEXT NOT NULL REFERENCES skill_definitions(level) DEFAULT 'B',
    priority INTEGER NOT NULL CHECK(priority >= 0) DEFAULT 0
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_area_tasks_unique ON area_tasks(area_code, task_code);
CREATE INDEX IF NOT EXISTS idx_area_tasks_area ON area_tasks(area_code);
CREATE INDEX IF NOT EXISTS idx_area_tasks_professional ON area_tasks(is_professional);

    level TEXT PRIMARY KEY CHECK(level IN ('S', 'A', 'B', 'C')),

    description TEXT,
    can_independent INTEGER NOT NULL CHECK(can_independent IN (0, 1)),
    score REAL NOT NULL CHECK(score >= 0 AND score <= 1)
);

-- 4.4 员工信息表
-- ============================================
CREATE TABLE IF NOT EXISTS employees (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,

CREATE INDEX IF NOT EXISTS idx_employees_main_area ON employees(main_area);
CREATE INDEX IF NOT EXISTS idx_employees_type ON employees(employee_type);
CREATE INDEX IF NOT EXISTS idx_employees_active ON employees(is_active);

    task_code TEXT NOT NULL,
    skill_level TEXT NOT NULL REFERENCES skill_definitions(level),

    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_agent_version ON agent_messages(version_id);
CREATE INDEX IF NOT EXISTS idx_agent_intent ON agent_messages(intent);
CREATE INDEX IF NOT EXISTS idx_agent_created ON agent_messages(created_at);

-- ============================================
-- 4.14 修改原因字典表
-- ============================================
CREATE TABLE IF NOT EXISTS modified_reasons (
    code TEXT PRIMARY KEY,
    label TEXT NOT NULL,
    sort_order INTEGER NOT NULL DEFAULT 0
);

-- ============================================
-- 4.15 促销配置表
-- ============================================
CREATE TABLE IF NOT EXISTS promotions (
    id TEXT PRIMARY KEY,
    date TEXT NOT NULL,
    area_code TEXT NOT NULL REFERENCES areas(code) ON DELETE CASCADE,
    promotion_type TEXT NOT NULL CHECK(promotion_type IN ('high', 'low')) DEFAULT 'low',
    boost_factor REAL NOT NULL CHECK(boost_factor > 1.0) DEFAULT 1.0,
    description TEXT
);
CREATE INDEX IF NOT EXISTS idx_promotions_date_area ON promotions(date, area_code);

-- ============================================

CREATE TABLE IF NOT EXISTS hc_suggestions (
    id TEXT PRIMARY KEY,

    employee_type TEXT NOT NULL CHECK(employee_type IN ('regular', 'temporary')),
    current_count INTEGER NOT NULL,
    suggested_count INTEGER NOT NULL,
    delta INTEGER NOT NULL,
    reason TEXT,
    cost_before REAL NOT NULL,
    cost_after REAL NOT NULL,
    status TEXT NOT NULL CHECK(status IN ('pending', 'approved', 'rejected')) DEFAULT 'pending',
    created_at TEXT NOT NULL DEFAULT (datetime('now'))

CREATE INDEX IF NOT EXISTS idx_hc_suggestions_status ON hc_suggestions(status);
CREATE INDEX IF NOT EXISTS idx_hc_suggestions_area ON hc_suggestions(area_code);
