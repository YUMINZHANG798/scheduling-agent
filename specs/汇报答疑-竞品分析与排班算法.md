# 汇报答疑：竞品分析与排班算法说明

> 适用场景：Demo 汇报、答辩问答、产品方案说明。  
> 核心口径：本 Demo 不是单纯做一个“排班表生成器”，而是验证“面向生鲜商超的、可解释、可约束、可降本的智能排班 Agent”。

## 1. 竞品分析：我们和现有系统相比有什么优势

### 1.1 一句话回答

现有劳动力管理系统普遍已经具备考勤、排班、预测、合规和报表能力，但大多偏通用 WFM 平台；我们的优势在于更贴近生鲜商超的业务场景，把“需求预测、专业师傅保护、正式工区域固定、小时工弹性补位、请假审批、重排冻结、降本冗余识别、Agent 解释”放在同一个闭环里，让排班负责人不仅能生成班表，还能知道为什么这样排、哪里不能请假、哪里存在人力冗余。

### 1.2 主要竞品类型

| 类型 | 代表产品 | 常见能力 | 对我们的启发 |
| --- | --- | --- | --- |
| 国际 WFM / HCM 平台 | UKG、Legion、Deputy、Quinyx | AI 预测、自动排班、员工偏好、合规规则、成本控制、跨门店/多行业支持 | 证明“需求预测 + 自动排班 + 劳动力优化”是成熟方向 |
| 国内劳动力管理平台 | 盖雅工场、喔趣、北森等 | 智能排班、考勤、工时、薪酬、人效分析、复杂规则配置 | 证明国内企业更关注工时、考勤、薪酬、人效闭环 |
| 通用协同/考勤工具 | 钉钉、企业微信生态应用等 | 排班、打卡、请假、审批、基础统计 | 适合轻量管理，但复杂预测和优化能力较弱 |
| 门店自建表格/人工经验 | Excel、微信群、手工模板 | 灵活、成本低、上手快 | 目前很多门店真实现状，但不可追溯、不可优化、依赖个人经验 |

### 1.3 我们的差异化优势

#### 1. 更强的行业场景贴合度

竞品通常强调通用劳动力管理：适用于零售、餐饮、制造、物流、医疗等多个行业。我们的 Demo 直接聚焦生鲜商超，把水产、肉类、果蔬、收银、补货五类区域拆开处理，并单独建模“杀鱼、切肉”等专业任务。

汇报时可以这样说：

> 通用 WFM 系统解决的是“把人排到班上”；我们解决的是“在生鲜商超里，哪些岗位必须由专业师傅覆盖，哪些高峰可以用小时工弹性补位，哪些正式工其实略有冗余”。

#### 2. 不是黑盒排班，而是可解释排班

本系统会把需求因素、排班原因和风险原因暴露出来，例如历史销售/客流、线上订单、天气、周末、促销、晚高峰、专业岗覆盖、区域保底等。右侧 Agent 还可以继续追问：“为什么周五果蔬区排这么多人？”“某个员工为什么不能请假？”“当前是否有人力冗余？”

优势：

- 店长能理解结果，降低对算法的不信任。
- 排班负责人能把系统结果转化为管理语言。
- 管理层能追溯用工成本变化的原因。

#### 3. 专业师傅保护是硬约束，不是普通技能标签

生鲜商超中，水产杀鱼、肉类切肉这类岗位不能简单用“人数够了”判断。系统会检查员工技能等级，要求专业任务必须由具备 S/A 级技能的正式工覆盖；如果同一天某区域专业师傅都申请休假，系统会拒绝请假并给出原因。

这点区别于通用排班工具中常见的“技能偏好”或“技能匹配”，我们把专业岗当成业务连续性的硬约束。

#### 4. 请假审批和重排逻辑更符合真实门店

当前 Demo 已实现：

- 员工至少提前一天申请休假。
- 系统判断是否会造成高峰期人员不足。
- 系统判断是否会导致某区域专业师傅不足。
- 申请失败时保留原班表，并给出失败原因。
- 申请成功后自动从休假日开始重排，已经过去的日期不变。

这个逻辑适合汇报时强调：

> 我们不是简单记录请假，而是先判断这一天业务能不能正常开展；排班系统要服务经营，而不是只做流程审批。

#### 5. 可以识别人力略微冗余，服务降本目标

很多排班系统会倾向于“把所有员工都排上”，这样看起来班表很满，但无法回答“公司是否人多了”。本 Demo 现在采用“刚性需求覆盖 + 少量安全缓冲”的策略：在保证专业岗、区域保底和高峰覆盖的前提下，允许少量正式工本周未被安排，从而暴露当前编制略高于业务刚需。

汇报口径：

> 如果系统每次把 80 名正式工全部排满，只能证明排班表很忙，不能证明编制合理。我们现在让算法保留少量未安排人员，用来帮助企业发现可优化空间。

#### 6. Agent 不直接篡改排班，排班结果可控

本系统中，LLM/Agent 主要负责解释、问答和汇报组织，不直接绕过规则生成班表。真正的排班由确定性规则引擎完成，核心指标也由程序计算。

优势：

- 结果稳定，可复现。
- 不会因为大模型幻觉破坏专业岗、工时、请假等硬约束。
- 适合企业级系统落地和审计。

### 1.4 对比总结表

| 对比维度 | 通用 WFM / 排班系统 | 本 Demo 的优势 |
| --- | --- | --- |
| 业务颗粒度 | 多行业通用，通常按岗位/门店建模 | 针对生鲜商超区域、时段、专业师傅、小时工补位建模 |
| 专业岗位 | 多为技能匹配或偏好 | 专业师傅覆盖是硬约束 |
| 请假处理 | 常见为流程审批或可用性管理 | 结合预测需求、专业师傅、区域保底判断是否批准 |
| 解释能力 | 有报表和提示，但不一定能自然语言追问 | 右侧 Agent 可持续追问排班原因、需求因素、风险和冗余 |
| 降本视角 | 多关注工时、预算、合规 | 直接显示正式工未安排人数，辅助判断轻微冗余 |
| Demo 可控性 | 商业系统能力完整但难演示内部逻辑 | 本项目可展示从数据、算法、排班到解释的完整链路 |

## 2. 排班算法是怎样的

### 2.1 一句话回答

当前 Demo 使用“需求预测 + 规则约束 + 启发式评分”的确定性排班算法：先根据历史销售、客流、线上订单、天气、节假日、促销等因素预测每个区域、每个时段需要多少人；再在专业师傅、正式工固定区域、小时工弹性补位、请假、工时上限等约束下生成班表；最后计算 KPI、风险和解释。未来上线版本会升级为“预测模型 + 约束优化 + 滚动重排 + 人效学习”的企业级算法体系。

## 3. 当前 Demo 中的排班算法

### 3.1 输入数据

当前 Demo 的算法输入包括：

- 员工数据：员工 ID、姓名、正式工/小时工、所属区域、默认班次、周工时上限、是否专业岗。
- 技能数据：员工在不同区域、不同任务上的技能等级。
- 区域与任务：水产、肉类、果蔬、收银、补货；杀鱼、切肉、称重、补货、收银等任务。
- 历史经营数据：历史销售、历史客流、线上订单。
- 外部因素：天气、节假日、促销、周末、会员日等。
- 员工休假申请：每周休假日、申请时间、审批结果。

### 3.2 第一步：需求预测

系统先生成未来一周的“需求结果”，粒度是：

> 日期 × 小时时段 × 区域 × 任务

每条需求会计算：

- 需求分 `demand_score`
- 需要总人数 `required_count`
- 专业师傅需求 `professional_required_count`
- 正式工需求 `regular_required_count`
- 小时工需求 `temporary_required_count`
- 影响因素 `demand_factors`
- 优先级 `priority`
- 置信度 `confidence`

当前 Demo 的需求分计算逻辑是加权启发式：

```text
需求分 = 客流分量 + 销售分量 + 线上订单分量 + 时段分量
再叠加周末、天气、促销、周五晚高峰等因子
```

例如：

- 周五晚高峰：提高晚间需求。
- 下雨：提升线上拣货和到店集中度。
- 周末：提升家庭采购类需求。
- 果蔬会员日、肉类特惠：提升对应区域需求。

### 3.3 第二步：确定正式工核心排班池

当前系统不会默认把所有正式工都排满，而是根据一周需求确定每个区域的“核心排班池”。

原则：

- 专业区先保留专业师傅。
- 高峰越多，核心池越大。
- 周末、促销、雨天、团购等外部因素会增加缓冲。
- 仍允许少量正式工不被安排，用于识别当前编制略微冗余。

当前 Demo 的效果是：

- 在强需求周，正式工可能安排约 77 / 80 人。
- 在平峰周，正式工可能安排约 72 / 80 人。
- 专业岗覆盖率仍保持 100%。
- 区域保底达成率仍保持 100%。

汇报时可以强调：

> 这不是算法排不满，而是算法有意不排满。它要回答的不只是“怎么排班”，还要回答“现有人是不是略多了”。

### 3.4 第三步：正式工排班

正式工排班遵循几个硬规则：

- 正式工固定在所属区域，不跨区。
- 专业师傅优先覆盖专业任务。
- 每个区域要满足保底正式工人数。
- 员工当天已批准休假则不参与当天排班。
- 班次覆盖当前支持：
  - 早班：08:00-16:00
  - 午间班：10:00-18:00
  - 晚班：14:00-22:00
  - 两头班：08:00-12:00 + 17:00-21:00

其中午间班是为了覆盖 12:00-14:00 的实际高峰，避免“人够但时段不覆盖”的问题。

### 3.5 第四步：小时工弹性补位

小时工用于跨区域补位，但不承担专业任务。

补位逻辑：

- 找出高优先级且非专业任务的需求。
- 判断当前正式工覆盖是否不足。
- 从小时工池中选择具备对应技能、未超工时、当天未冲突的候选人。
- 按技能等级、区域熟悉度、已用工时计算候选评分。
- 根据需求时段选择小时工班次：
  - 午前/午间需求：10:00-14:00
  - 下午需求：14:00-18:00
  - 晚间需求：18:00-22:00 或 17:00-22:00

### 3.6 第五步：请假审批与局部重排

请假不是简单保存，而是模拟判断。

审批规则：

- 员工必须至少提前一天申请。
- 如果请假会导致当天某区域正式工不足，则拒绝。
- 如果请假会导致专业师傅不足，则拒绝。
- 如果模拟重排后专业岗或区域保底不能满足，则拒绝。
- 请假成功后自动重排，但只从请假的日期开始重排，已经过去的日期不变。

这个逻辑贴近真实门店：已经执行过的班表需要留痕，不应该被后续请假重写。

### 3.7 第六步：风险检测与 KPI

系统会检测：

- 专业岗缺口。
- 高峰缺口。
- 区域保底是否满足。
- 小时工利用率。
- 人工干预率。
- 当前正式工是否存在未安排人数。

这些指标用于支持两个目标：

1. 确保业务正常开展。
2. 识别人力成本优化空间。

### 3.8 当前算法特点

| 特点 | 说明 |
| --- | --- |
| 确定性 | 同一输入可复现，适合 Demo 和企业审计 |
| 可解释 | 每个需求和排班都有因素、规则和解释 |
| 规则优先 | 专业岗、请假、区域保底、工时上限不会被 LLM 破坏 |
| 轻量快速 | 不依赖大型优化器，单门店一周可快速生成 |
| 可扩展 | 后续可替换需求预测模型和优化求解器 |

## 4. 未来完善上线后的算法

### 4.1 总体方向

上线版本建议升级为：

```text
多源数据接入
→ 机器学习需求预测
→ 约束优化排班
→ 实时滚动重排
→ 人效与成本闭环学习
→ Agent 解释与决策辅助
```

### 4.2 需求预测升级

当前 Demo 使用启发式加权规则，上线后可以升级为机器学习预测模型。

可接入数据：

- POS 销售流水
- 客流摄像头/客流计
- 线上订单与拣货单
- 商品品类与 SKU
- 促销计划
- 天气预报
- 节假日与商圈活动
- 历史排班与实际考勤
- 实际缺货、排队、投诉、等待时长等服务指标

模型方向：

- 时间序列模型：Prophet、ARIMA、LightGBM、XGBoost。
- 深度学习模型：TFT、LSTM、Transformer 时序模型。
- 分层预测：门店级、区域级、任务级、15/30/60 分钟级。
- 不确定性预测：输出 P50/P80/P95 需求，给排班留安全余量。

上线后不只预测“需要几个人”，还要预测：

- 哪个区域忙。
- 哪个任务忙。
- 忙到什么程度。
- 需要正式工还是小时工。
- 需要什么技能等级。
- 预测置信度是多少。

### 4.3 排班优化升级

当前 Demo 是规则 + 启发式评分，上线后可以引入约束优化。

可选技术：

- MILP 混合整数规划
- CP-SAT 约束规划
- OR-Tools
- 启发式搜索 / 局部搜索
- 遗传算法 / 模拟退火
- 多目标优化

优化目标可以包括：

```text
最小化总人力成本
+ 最小化高峰缺口
+ 最小化专业岗风险
+ 最小化加班
+ 最小化员工偏好违背
+ 最小化临时调整次数
+ 最大化公平性
+ 最大化服务水平
```

约束条件可以包括：

- 专业岗位必须由合格师傅覆盖。
- 区域最低正式工保底。
- 员工工时上限。
- 连续工作天数限制。
- 两班间休息时间。
- 请假、培训、不可用时段。
- 劳动法规与企业制度。
- 员工偏好和公平性。
- 预算上限。

### 4.4 重排算法升级

上线后重排要支持实时业务变化：

- 天气突变。
- 临时促销。
- 员工临时请假。
- 线上订单突然增长。
- 某区域排队过长。
- 某员工迟到或缺勤。

重排原则：

- 已经执行的班表冻结。
- 即将开始的班次尽量少改。
- 优先用小时工和机动人员补位。
- 必要时才调整正式工。
- 每次调整都记录原因和影响。

可采用滚动优化：

```text
每 15/30/60 分钟更新需求预测
→ 只优化未来可调整窗口
→ 保留已执行和已通知员工的班次
→ 输出变更影响和通知建议
```

### 4.5 人效与成本闭环

上线后可以把排班结果和实际经营结果闭环：

- 预测需求 vs 实际客流/订单。
- 计划排班 vs 实际出勤。
- 排班覆盖 vs 排队等待/投诉/销售损失。
- 人力成本 vs 销售额/毛利。
- 员工技能成长 vs 可排岗位变化。
- 店长人工修改原因 vs 算法规则优化。

最终输出不只是班表，而是编制建议：

- 某区域是否正式工略多。
- 某区域是否专业师傅不足。
- 是否应该增加小时工池。
- 哪些员工适合培训成专业师傅。
- 哪些时段可以减少排班。

### 4.6 Agent 上线后的角色

上线版本中，Agent 不应该直接替代算法，而应该成为算法解释层和决策交互层。

Agent 可以做：

- 解释为什么这样排。
- 解释为什么不能请假。
- 解释为什么建议减少/增加某类员工。
- 帮店长用自然语言查询班表。
- 汇总本周人力成本变化。
- 生成给区域负责人或 HR 的说明。

Agent 不应该做：

- 绕过硬约束直接改班。
- 生成不可审计的黑盒班表。
- 在没有数据依据时编造需求原因。

## 5. 汇报时可以使用的短回答

### Q1：和竞品相比，我们有什么优势？

> 竞品大多是通用劳动力管理平台，能力很完整，但往往偏“通用排班、考勤和工时管理”。我们的 Demo 更聚焦生鲜商超：把专业师傅、区域保底、小时工跨区补位、天气/促销/周末高峰、请假审批、局部重排和人力冗余识别整合在一个闭环里。尤其是 Agent 能解释每一次排班决策，让店长知道为什么这样排、为什么某些假不能批，以及当前编制是否略有冗余。

### Q2：当前算法是不是 AI？

> 当前 Demo 的排班结果不是让大模型直接生成，而是由确定性规则引擎生成。需求预测用历史销售、客流、线上订单、天气、节假日、促销等因素计算；排班阶段用规则和评分保障专业岗、区域保底、工时、请假和小时工补位。LLM 主要负责解释和追问，不直接破坏排班规则。这种设计更适合企业级落地，因为结果稳定、可解释、可审计。

### Q3：为什么不是把所有员工都排满？

> 如果每次都把所有正式工排满，系统就无法判断企业是否人力略微冗余。现在算法采用“刚性需求覆盖 + 小缓冲”的策略，在保证专业岗和高峰覆盖的前提下，允许少量正式工本周未被安排。这样管理层可以看到当前编制比业务刚需多一点，从而评估是否需要降低人力成本。

### Q4：未来上线后算法会怎么升级？

> 未来会上线机器学习需求预测和约束优化排班。需求侧会接入 POS、客流、线上订单、天气、节假日、促销、商圈活动等数据，预测到区域、任务、15/30 分钟粒度；排班侧会使用约束规划或混合整数规划，在专业岗、工时、请假、合规、预算、公平性等约束下做多目标优化；运行中还会支持滚动重排和人效闭环，让系统持续学习店长修改和真实经营结果。
