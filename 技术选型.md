# 智慧排班 Agent 技术选型文档（展示版 v1）

## 1. 目标与约束

### 1.1 目标

- 在 4-8 周内做出可演示且核心流程可运行的第一版系统。
- 覆盖 PRD 的 P0 主链路：
  - 数据导入
  - 需求预测
  - 自动排班
  - 合规校验
  - 店长审批发布
  - 员工查班/请假/换班
  - Agent 对话式调班与解释

### 1.2 展示版约束

- 优先可运行、可解释、可演示，不追求一次性最优算法精度。
- 采用单体优先、模块化设计，避免过早微服务化。
- 选择成熟组件，减少自研基础设施。

## 2. 选型原则

- `稳定优先`：选择生产验证充分、社区成熟的框架。
- `开发效率优先`：保证两到三人小团队也能快速推进。
- `可演进`：展示版架构可平滑扩展到生产版。
- `可观测`：核心链路必须有日志、监控和审计留痕。
- `可替换`：预测模型、优化引擎、LLM 供应商可替换。

## 4. 各模块选型与取舍

## 4.1 前端：Next.js 14 + TypeScript

### 为什么选

- 支持 SSR/CSR 混合，后台管理类界面开发效率高。
- TypeScript 能约束复杂排班对象结构，降低联调错误。
- 生态成熟，适合快速构建店长工作台和员工端页面。

### 不选项

- Vue/Nuxt：可行，但当前团队如 Python/TS 栈更统一，React 生态更易招聘。
- Flutter Web：对运营后台不具优势，组件生态与可维护性成本更高。

## 4.2 后端：FastAPI

### 为什么选

- 和预测、优化、规则模块同为 Python 生态，研发路径短。
- 自动 OpenAPI 文档，便于前后端联调和演示。
- 对异步和数据校验支持良好。

### 不选项

- Java/Spring：企业级强，但展示版开发速度不占优。
- Node.js/NestJS：也可行，但预测和优化核心在 Python，跨语言复杂度上升。

## 4.3 数据库：PostgreSQL

### 为什么选

- 事务一致性强，适合排班发布、审批、审计等核心业务。
- JSONB 支持规则配置和半结构化数据。
- 可选 pgvector，后续支持知识检索与解释增强。

### 不选项

- MongoDB：事务和复杂查询不如 PostgreSQL 直接。
- MySQL：可行，但 JSON/扩展能力略弱于 PostgreSQL 生态。

## 4.4 排班优化：OR-Tools CP-SAT

### 为什么选

- 对排班类整数约束问题成熟，表达力强。
- 支持硬约束和软约束，适合“可行解优先，再优化公平性和偏好”。
- Python API 友好，方便快速迭代规则。

### 不选项

- 自研启发式算法：前期可快，但很快会陷入规则爆炸和质量不可控。
- 商业求解器（如 Gurobi）：性能强但商业成本高，展示版不必要。

## 4.5 预测模块：Prophet + LightGBM

### 为什么选

- Prophet 可快速上线有节假日季节性的基线预测。
- LightGBM 作为增强模型，逐步引入天气/促销/订单等特征。
- 展示版可先用“规则+基线模型”，保证可用和可解释。

### 不选项

- 端到端深度学习（LSTM/Transformer）：数据和调参成本高，展示阶段性价比低。

## 4.6 LLM Agent：OpenAI API + 工具调用

### 为什么选

- 通过函数调用将“解释、建议、调班意图”绑定到可执行工具，避免幻觉直接落库。
- 快速实现店长自然语言调班和冲突解释场景。
- 可加人审流程，确保所有最终变更受控。

### 安全策略

- LLM 仅生成建议，不直接写入正式班表。
- 所有变更必须经过规则校验和店长确认。
- 关键操作写审计日志。

## 5. 展示版核心能力映射

| PRD 功能 | 展示版实现策略 |
| --- | --- |
| 门店分时段需求预测 | 基线模型 + 节假日因子 + 店长手动修正 |
| 自动生成周班表 | OR-Tools 硬约束求可行解 + 软约束优化 |
| 合规校验 | 可配置规则集（工时、休息间隔、提前通知） |
| 任务级分配 | 班次内任务段分配（收银/补货/拣货） |
| 店长审批 | 审批流 + 发布按钮 + 版本留痕 |
| 员工换班 | 候选人自动筛选 + 审批 |
| Agent 解释 | “为什么这样排”“为什么不可行”可解释输出 |

## 7. 开发与部署选型

### 7.1 环境分层

- `local`：开发联调，Docker Compose 一键拉起。
- `staging`：演示环境，接近生产配置。
- `prod`：后续上线环境（展示版可不启用）。

### 7.2 CI/CD

- GitHub Actions（或同类）执行：
  - 单元测试
  - API contract 检查
  - 镜像构建
  - 部署到 staging

### 7.3 配置管理

- `12-factor` 风格，全部通过环境变量注入。
- 密钥统一走 Secret 管理，不写入仓库。

## 8. 非功能目标（展示版）

- 可用性：演示窗口期间服务可用性 >= 99%。
- 性能：
  - 单店周排班求解 <= 15 秒
  - 50 店批量排班 <= 10 分钟
- 安全：
  - RBAC 生效
  - 关键操作审计全量留痕
- 可观测：
  - 请求日志、任务日志、错误告警可追踪

## 9. 风险与替代方案

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 预测准确率不足 | 排班体验下降 | 允许店长修正，先保证可行与稳定 |
| 约束过多导致无解 | 无法排班 | 输出冲突解释，给出最小放松建议 |
| LLM 误导建议 | 业务风险 | LLM 不直接落库，规则引擎兜底 |
| 集成数据缺失 | 链路不完整 | 先用 CSV 导入 + 模拟数据演示 |

## 10. 实施结论

展示版建议采用：

- 前端：`Next.js + TypeScript`
- 后端：`FastAPI + Celery`
- 存储：`PostgreSQL + Redis`
- 算法：`Prophet/LightGBM + OR-Tools`
- Agent：`OpenAI API 工具调用`
- 部署：`Docker Compose`

该组合能在最短周期内跑通 PRD 核心功能，并保留后续向企业级版本演进的空间。
